# 缺陷管理最佳实践指南

## 缺陷信息编写规范

### 1. 缺陷标题 (Title)

**目的**: 快速识别问题的本质

**好的例子**:
- ✓ "登录页面验证码显示为黑色方块"
- ✓ "用户列表页面API接口返回超时"
- ✓ "数据导出功能在大数据量时崩溃"

**不好的例子**:
- ✗ "系统有问题"
- ✗ "bug"
- ✗ "请查看"

**编写建议**:
- 准确、清晰、简洁（40字以内）
- 包含功能模块 + 具体问题
- 避免主观用语（"太慢"、"不好"等）
- 避免重复字符或特殊符号

### 2. 环境 (Environment)

标准环境值：
- **开发** - 本地开发环境
- **测试** - QA测试环境
- **预发** - 预发布环境（最接近线上）
- **线上** - 生产环境
- **其他** - 特殊环境

**重要**: P0级别的线上环境缺陷应立即上报。

### 3. 预期结果 (Expected Result)

**定义**: 功能在正确情况下应该表现的行为

**好的例子**:
- 用户输入正确密码后，系统应在3秒内完成登录并跳转到首页
- 列表应按降序显示最近10条数据
- 导出文件应包含所有选中的记录，格式为xlsx

**编写建议**:
- 明确、可量化
- 包括时间要求（如适用）
- 包括数量或质量指标

### 4. 实际结果 (Actual Result)

**定义**: 当前系统实际表现出的行为

**好的例子**:
- 登录失败，返回错误代码500
- 列表显示为乱码，无法读取数据
- 导出的文件只包含部分数据

**编写建议**:
- 客观描述观察到的现象
- 包括错误信息或错误代码
- 如果有错误日志，引用关键错误行

### 5. 重现步骤 (Reproduction Steps)

**定义**: 从头开始复现问题所需的精确步骤

**好的例子**:
```
1. 打开 https://test.example.com/login
2. 输入用户名: test@example.com
3. 输入密码: Test@123
4. 点击"登录"按钮
5. 观察结果：显示黑色验证码方块
6. 尝试刷新页面，验证码仍为黑色
```

**编写建议**:
- 编号并详细列出每一步
- 包括具体的输入值（不要用"xxx"占位）
- 明确指出观察点
- 提供足够的细节让任何人都能复现

### 6. 优先级 (Priority)

| 级别 | 定义 | 处理时间 | 示例 |
|------|------|---------|------|
| P0 | 严重，系统不可用 | 立即 | 系统完全崩溃、数据丢失、安全漏洞 |
| P1 | 重要，主要功能受阻 | 24小时 | 核心功能无法使用、重要模块故障 |
| P2 | 中等，功能有缺陷 | 1周 | 界面显示错误、部分功能不正常 |
| P3 | 轻微，边界问题 | 2周 | 拼写错误、样式问题、罕见场景下的故障 |

**以下信息可帮助判断优先级**:
- 影响的用户数量
- 问题发生的频率
- 是否影响收入或品牌
- 是否有可用的解决方案

### 7. 重现频率 (Frequency)

| 值 | 含义 | 处理建议 |
|----|------|---------|
| 100%稳定复现 | 每次都能复现 | 优先处理，定位快速 |
| 90%以上 | 大部分情况复现 | 较容易定位，需注意环境因素 |
| 50-90% | 需要多次尝试 | 可能涉及并发或竞态条件 |
| 50%以下 | 间歇性出现 | 可能涉及缓存或异步问题，定位困难 |

### 8. 附件 (Attachments)

**建议提供以下附件**:
- **截图** - 错误现象的可视化证明
- **录屏** - 复杂交互流程的演示
- **日志文件** - 浏览器控制台或服务器日志
- **配置文件** - 系统配置或环境信息
- **测试数据** - 必要的样本数据（脱敏处理）

**附件命名规范**:
```
模块_问题_日期.扩展名
例: login_blackscreen_20260416.png
```

```
重现步骤:
```


## 常见错误

### 错误1: 标题过于模糊

❌ "登录不了"
✓ "输入错误密码后登录页面返回HTTP 500错误"

### 错误2: 重现步骤困难或不清楚

❌ "点击某处后，系统就有问题了"
✓ [见上面详细的步骤说明]

### 错误3: 混淆预期和实际结果

❌ "页面显示错误" （既是预期也是实际）
✓ "预期: 显示蓝色按钮; 实际: 显示红色按钮"

### 错误4: 关键信息遗漏

❌ 没有提供附件，说不清问题
❌ 没有提供浏览器版本
❌ 没有说明是否能稳定复现

## 缺陷生命周期

```
新建 → 已分配 → 处理中 → 已修复 → 验证中 → 已关闭
  ↓
已拒绝
```

- **新建**: 刚创建的缺陷
- **已分配**: 已指派给开发人员
- **处理中**: 开发人员正在处理
- **已修复**: 开发人员修复完成，提交测试
- **验证中**: 测试人员验证修复效果
- **已关闭**: 修复验证通过，缺陷关闭
- **已拒绝**: 经讨论认为不是缺陷，被拒绝

## 关于缺陷的常见问题

### Q: 在什么情况下应该创建缺陷？

A: 以下情况应创建缺陷：
- 功能不按预期工作
- 系统出现错误信息
- 性能明显下降
- 安全问题或数据问题
- 用户体验问题

### Q: 如何判断优先级？

A: 考虑以下因素：
- 有多少用户受影响
- 业务影响程度
- 是否有可用的替代方案
- 发生频率

### Q: 重复的缺陷应该怎么处理？

A: 
- 避免创建完全相同的缺陷
- 如果发现重复，备注原始缺陷ID
- 可以在原始缺陷中添加评论补充信息

### Q: 缺陷多久应该关闭？

A:
- P0: 24小时内必须处理
- P1: 3-5天内处理
- P2: 1-2周内处理
- P3: 2-4周内处理
